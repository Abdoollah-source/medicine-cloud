Medicine Cloud — Project Overview

How the notes are made today, and the plan to turn it into a real web app.

1. What this project actually is

"Medicine Cloud" is a personal system for turning messy lecture notes — PDFs, photos of handwriting, screenshots, voice messages, and scattered links — into clean, good-looking study notes that can be read on a phone or computer. It also links important medical terms (diseases, drugs, organisms, etc.) straight to their Wikipedia articles, and embeds media like YouTube videos or Telegram posts right inside the note.

Right now this is done manually, one note at a time, using two files. The goal of this project is to turn that manual process into an automated, always-available website that a small group of people (the project owner and a few friends/classmates) can log into, use, and save their notes to permanently.

2. How it works today (the manual process)

File 1 — medicine-cloud-json-extraction-prompt.md

This is a long, very strict instruction sheet that gets pasted into an AI chat, along with the raw lecture material (PDFs, photos, screenshots, links). It tells the AI to act like a careful data-entry machine, not a creative writer. The rules include:

  • Never invent facts, numbers, or explanations that aren't in the source material.
  • Only fix spelling, grammar, and OCR garbage — don't rewrite or shorten the original meaning.
  • Keep each language exactly where it appears (the notes mix English and Arabic) — never translate unless explicitly told to.
  • Only use a fixed, limited set of HTML tags (paragraphs, headings, lists, tables, bold/italic, and special colored links for medical terms).
  • Turn any media links (YouTube, Telegram, images, audio, video) into specific placeholder tags like [[YT:videoID]] or [[IMAGE:url]] that the website later understands.
  • Output only one clean JSON object, wrapped in a JSON array — nothing else.

The result of this step is a single block of JSON text containing the note's title, language, subject, date, and the fully formatted content.

File 2 — medicine-cloud-note_claude_.html

This is a ready-made, self-contained webpage with all its design (fonts, colors, light/dark mode, Arabic/English text direction) already built in. It has one empty spot near the bottom — a <script id="note-data" type="application/json"> tag — where the JSON produced in Step 1 gets pasted by hand.

Once the JSON is pasted in and the file is saved, opening it in a browser automatically:

  • Displays the title, subject, unit, and date nicely.
  • Renders the content with the right fonts and text direction for the language.
  • Converts every [[YT:...]], [[IMAGE:...]], [[TG_POST:...]], etc. placeholder into a real embedded video, image, or link-out card.
  • Turns colored medical terms into clickable Wikipedia links.
  • Lets the reader switch between light/dark theme and (if a translated version was generated) between languages.
In short, the manual workflow today is:
  1. Gather the lecture material (PDFs, photos, links).
  2. Paste the material + the prompt file into an AI chat.
  3. Copy the JSON the AI produces.
  4. Paste that JSON into the HTML file, in the exact right spot.
  5. Save the file and open it — the note is now readable and shareable.

3. The problem with the manual process

  • Every note is its own separate HTML file — nothing is centralized or searchable.
  • Pasting JSON by hand into a script tag is fragile — one broken quote or comma breaks the whole page.
  • There's no way to save a library of notes, edit them later, or apply a different look to an old note without redoing the paste-and-save process.
  • Sharing means sending an HTML file around — no proper website, no PDF export, no accounts.

4. The plan — turning this into a real web app

The goal is a website where the same JSON gets pasted into a text box (instead of a file), a template and color palette get picked, and the note is instantly previewed, saved, and exportable — with zero cost and no risk of the service disappearing or throttling due to normal, small-scale use.

Who can use it

Not a public tool — just the project owner and a small group of friends/classmates. Each person gets their own account and their own private notes by default, with the option to share an individual note with a link when they want to.

How people log in

Google Sign-In was chosen as the main method — one tap, no password to create, remember, or leak. (A password-free email "magic link" option can exist as a backup for anyone who prefers it.) This also solves the earlier concern about a shared password: if there's no shared secret, there's nothing for someone to leak or misuse to damage everyone else's notes.

The two free-tier "gotchas" and how they're handled

Supabase pauses a project after about a week with no activity. Fixed with a free automated check-in (a scheduled GitHub Actions job) every few days, so the database never goes to sleep.

Supabase's free email sending has limits. Solved by using Google Sign-In instead of emailed login links, which has no such limit.

5. Why this approach was chosen over the alternatives

  • Client-side-only PDF generation (screenshotting the page in the browser) was considered, but rejected — it's known to cause exactly the problems this project wants to avoid: text getting cut off at page breaks and layouts shifting.
  • Using a Telegram bot as the main database was considered (an idea worth crediting to the project owner). It turned out to be a poor fit as the primary storage, because Telegram's bot system has no clean way to list or search everything that's been saved — you'd still need a separate index somewhere, which defeats the purpose. It remains a good idea as a free backup mirror for saved notes, just not the main database.
  • A shared password for the whole group was considered and rejected for the reason the project owner raised: a single leaked password could let someone damage everyone's notes. Individual accounts with proper permissions solve this at no extra cost.

6. Where things stand

Planning complete

The idea, the manual workflow, the requirements, and the technical architecture have all been discussed and agreed on. The next step is building the actual project — starting with the core template-rendering system (since everything else depends on it), then adding accounts and saved-note storage, then the PDF export function.
