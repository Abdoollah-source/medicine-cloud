# Medicine Cloud — Constitution

> **This document is law.** Every spec, every feature, every line of code must be checked against
> this constitution before it is built. If a spec contradicts any article here, the spec must be
> revised — not the constitution. The constitution may only be amended by the project owner with
> explicit intent.

---

## Preamble

Medicine Cloud is a personal, invite-only web application that transforms messy medical lecture
material — PDFs, handwritten photos, screenshots, voice messages, links — into clean, beautiful,
permanent study notes. It automates and centralises what was previously a fully manual two-file
workflow (an AI extraction prompt + a standalone HTML note page) so that a small, trusted group of
people can log in, create, save, browse, and export their notes without technical friction.

The project must remain **zero-cost at normal scale**, **always available**, **private by default**,
and **impossible to silently break** by a single bad action.

---

## Article 1 — Users & Access

### 1.1 Invite-Only
Medicine Cloud is **not a public product**. There is no self-registration. A new account can only
be created when the project owner explicitly grants access. The login system must enforce this: a
valid Google sign-in from an uninvited email must be rejected with a clear "access not granted"
message, never silently accepted.

### 1.2 Authentication Method
**Google Sign-In (OAuth 2.0 via Supabase Auth) is the sole primary authentication method.**
No passwords are created, stored, or emailed. A password-free email magic-link may exist as a
secondary fallback for people who cannot use Google, but it must also be restricted to the allowlist.

### 1.3 Account Isolation
Every user's notes are private to that user by default. No user can ever read, modify, or delete
another user's notes unless the note owner has explicitly shared that note via a share link. There
are no "admin views" into other users' vaults in the production UI.

### 1.4 No Shared Secrets
There is no shared password, no shared API key, and no shared session that, if leaked, would
compromise other users' data. Each account is independently credentialed.

## Article 2 — Data & Storage

### 2.1 Primary Database
**Supabase (PostgreSQL)** is the sole primary data store. Notes, user records, share tokens, and
soft-delete data all live here.

### 2.2 Free-Tier Sustainability
The Supabase free tier pauses a project after roughly a week of inactivity. A **GitHub Actions
keep-alive job** (a lightweight scheduled HTTP ping) must be set up and remain active for the
lifetime of the project to prevent this. This is a hard infrastructure requirement, not optional.

### 2.3 Soft Delete — Nothing is Permanently Destroyed
When a user deletes a note, it must be **soft-deleted** (a `deleted_at` timestamp is set; the row
remains in the database). Hard (permanent) deletion must never happen automatically and must require
an explicit, confirmed secondary action by the user ("Permanently erase"). Soft-deleted notes must
be recoverable by the owner for a minimum of 30 days.

### 2.4 Row-Level Security (RLS)
**Supabase Row-Level Security must be enabled on every table that holds user data.** No table may
be left open to the public (anonymous or authenticated) without an explicit policy that limits
access to the row's owner. This is enforced at the database level, not just the application level.

### 2.5 Data Schema — Canonical Note Object

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key, generated by DB |
| `user_id` | UUID | Foreign key to `auth.users` |
| `title` | text | Required |
| `lang` | text | `"en"` or `"ar"`, required |
| `subject` | text | Nullable |
| `unit` | text | Nullable |
| `date` | text | Nullable |
| `content` | text | Raw HTML string (from extraction prompt) |
| `alt` | JSONB | Bilingual alt object, nullable |
| `template_id` | text | Template key, default `"classic"` |
| `is_public` | boolean | Default `false` |
| `share_token` | UUID | Nullable, generated on first share |
| `deleted_at` | timestamptz | Null = not deleted |
| `created_at` | timestamptz | Server default `now()` |
| `updated_at` | timestamptz | Updated on every write |

### 2.6 Telegram as Backup Mirror (Optional, Non-Primary)
A Telegram bot may be used as a **backup mirror** (secondary copy sent to a private channel). It
is never the primary database. If Telegram is unavailable, the app continues to work normally.

## Article 3 — The Note Format (The Sacred Contract)

### 3.1 JSON Schema
A note's content is always transported and stored as one JSON object conforming exactly to:

```json
{
  "title": "string — required",
  "lang": "en|ar — required",
  "subject": "string — optional, omit key if not applicable",
  "unit": "string — optional, omit key if not applicable",
  "date": "string — optional, omit key if not applicable",
  "content": "string of inline HTML — required",
  "alt": {
    "title": "string",
    "lang": "en|ar",
    "content": "string of inline HTML"
  }
}
```

The `alt` field is only present when a bilingual version was explicitly requested. No other fields
may exist at the top level of this object.

### 3.2 Allowed HTML Tags in content
The `content` string is injected via `innerHTML`. Only these tags are valid:
- `<p>` — paragraphs
- `<h2>`, `<h3>` — section and subsection headings
- `<ul>`, `<ol>`, `<li>` — lists
- `<strong>`, `<em>` — bold and italic
- `<table>`, `<tr>`, `<th>`, `<td>` — tables
- `<sup>`, `<sub>` — superscript and subscript
- `<a href='...' class='term-CATEGORY' target='_blank' rel='noopener noreferrer'>` — term links

Explicitly forbidden: `<script>`, `<style>`, `<iframe>`, `<form>`, `<input>`, `<img>`, `<div>`,
`<span>`, or any other tag not listed above. All HTML attribute values inside `content` must use
single quotes (not double quotes).

### 3.3 Term Link Categories (exactly five)

| Class | Category |
|---|---|
| `term-microbe` | Organisms, bacteria, viruses, fungi, parasites |
| `term-protein` | Proteins, enzymes, receptors, biomolecules |
| `term-pathology` | Diseases, syndromes, pathological processes |
| `term-anatomy` | Anatomical structures, organs, tissues |
| `term-pharma` | Drugs, medications, drug classes |

All links must point to **English Wikipedia only** (`https://en.wikipedia.org/wiki/...`).

### 3.4 Media Placeholder Tags
These appear as standalone lines in `content`, never nested inside a `<p>` or other tag:

| Tag | Purpose |
|---|---|
| `[[YT:VIDEO_ID]]` or `[[YT:VIDEO_ID\|Caption]]` | YouTube video embed |
| `[[TG_POST:channel/post_id]]` or `...\|Caption]]` | Telegram public post embed |
| `[[TG_AUDIO:full_url]]` or `...\|Caption]]` | Telegram voice/audio (link-out card) |
| `[[AUDIO:full_url]]` or `...\|Caption]]` | Direct audio file (native player) |
| `[[TG_IMAGE:full_url]]` or `...\|Caption]]` | Telegram image (link-out card) |
| `[[IMAGE:full_url]]` or `...\|Caption]]` | Direct hotlinkable image |
| `[[VIDEO:full_url]]` or `...\|Caption]]` | Direct video file |

Ampersands inside URLs must be written as `&amp;`. The renderer's `processMediaTags()` function
converts these placeholders into proper HTML embed markup.

### 3.5 Language & Direction
- `lang: "ar"` → note renders RTL, Arabic font stacks are applied.
- `lang: "en"` → note renders LTR, Latin font stacks are applied.
- Content must **never be auto-translated** without explicit user instruction.

## Article 4 — The Template System

### 4.1 Principle
A **template** is a visual skin (color tokens, fonts, layout tweaks) applied on top of the same
rendered note content. Switching templates **never alters the stored JSON or content HTML**.

### 4.2 Implementation
Templates are CSS variable overrides. The base stylesheet defines required tokens. Each template
provides its own values via a separate `.css` file loaded at template-switch time.

### 4.3 Required Design Tokens (all templates must define)
```
--paper, --paper-2                   background layers
--ink, --ink-soft, --ink-faint       text hierarchy
--gold, --gold-bright                accent / highlight
--rule, --rule-soft                  divider lines
--seal-bg, --shadow                  card surfaces
--grain-opacity                      texture overlay intensity
--color-microbe, --color-protein,
--color-pathology, --color-anatomy,
--color-pharma                       term link colors
--bg-term                            term link background tint
--font-display, --font-body,
--font-note                          Latin font stacks
--font-display-ar, --font-body-ar,
--font-note-ar                       Arabic font stacks
--ease, --ease-slow                  animation easing curves
```

### 4.4 Built-in Templates
- **`classic`** — the existing design from `medicine-cloud-note(claude).html`. The default.
  Must be pixel-identical to the original for users who depend on it.
- **One additional template** must ship at launch (e.g. `academic-dark`).

### 4.5 Light / Dark Mode
Every template must support both modes. OS preference is the default; users can override per
session. Preference persists in `localStorage` under the key `mc-theme`.

## Article 5 — Hosting & Infrastructure

### 5.1 Frontend
**Cloudflare Pages** (static hosting). No server-side rendering. No persistent server process.

### 5.2 Backend
- **Supabase** — database, auth, optional file storage.
- **Cloudflare Workers** (or equivalent free serverless) — only for PDF export.
  No other server-side logic is permitted.

### 5.3 Zero-Cost Constraint
Every infrastructure choice must fit within the free tier at expected scale. Any feature requiring
paid infrastructure must be flagged before implementation begins.

### 5.4 Renderer Independence
The note renderer must work entirely in the browser with no runtime API calls. Google Fonts CDN
link tags are permitted. Everything else is self-contained.

## Article 6 — Export

### 6.1 HTML Export (Browser-Side)
Generates a self-contained HTML file identical in appearance to the on-screen note:
- All CSS inlined or referenced via Google Fonts link tags.
- Note JSON embedded in `<script id="note-data" type="application/json">`.
- Renderer JS embedded inline.
- Selected template tokens applied.
- Generated entirely in the browser via Blob URL download. No server call.

### 6.2 PDF Export (Serverless Only)
Serverless function workflow:
1. Receives a render URL for the note.
2. Opens it in headless Chromium.
3. Calls native print-to-PDF.
4. Returns the PDF binary.

**Client-side screenshot-based PDF generation (html2canvas, jsPDF canvas, etc.) is explicitly
prohibited.** It produces cut-off text and broken layouts.

### 6.3 Template Fidelity
Both exports must use the template active when the export was triggered.

## Article 7 — Security

### 7.1 Private by Default
All notes start `is_public = false`. Sharing is always opt-in.

### 7.2 Share Tokens, Not Note IDs
Public share URLs use the `share_token` UUID. The internal `id` is never exposed in a user-visible
URL, preventing enumeration.

### 7.3 Content Sanitisation
The renderer must sanitise `content` using an allowlist-based HTML sanitiser before `innerHTML`
injection. Any tag or attribute not in the allowed set (§3.2) is stripped.

### 7.4 Secrets Handling
- Supabase URL + anon key: `.env`, injected at build time via environment variables.
- Supabase service-role key: **never in any frontend file**.
- `.env.example` (empty values) committed; `.env` (real secrets) in `.gitignore`.

### 7.5 No Sensitive Data in URLs
Auth tokens, internal note IDs, and user IDs must never appear in browser-visible URLs.

## Article 8 — UI / UX Principles

### 8.1 Performance
First meaningful content in under 2 seconds on a standard 4G connection. No heavy frameworks
unless explicitly approved. Vanilla HTML + CSS + JS is the default.

### 8.2 Mobile-First Responsive
Minimum viewport: 360 px wide. Every page must be fully usable on mobile.
Breakpoints: 360 px (mobile), 768 px (tablet), 1280 px (desktop).

### 8.3 Bidirectional Text
The UI chrome must support both LTR and RTL. Note content direction is set by the note's `lang`
field, independently of the app-level UI language. Both coexist on the same screen cleanly.

### 8.4 Accessibility
- Visible focus indicators on all interactive elements.
- Meaningful `alt` text on all images.
- Color is never the sole way information is conveyed.
- Target: WCAG 2.1 AA.

### 8.5 Reduced Motion
All animations and transitions must respect `@media (prefers-reduced-motion: reduce)`.

### 8.6 Explicit Error States
Every async operation must have a visible, user-friendly error state. Silent failures are
unacceptable. Error messages explain what went wrong and suggest a remedy.

### 8.7 Design Language — Neo-Brutalist System (THE LAW)

The application's entire visual identity is defined by **`design-tamplete.html`** located in
`MC-WEB_APP/`. Every page, every component, every interactive element must conform to this design
language exactly. Deviating from it is a constitutional violation.

#### Color Palette (canonical values — no substitutions)

| Token | Value | Usage |
|---|---|---|
| `--cream` | `#f4e9d8` | Page background, card surface |
| `--ink` | `#111111` | All text, borders, shadow layers, icon strokes |
| `--coral` | `#ff6f59` | Accent: final shadow offset on hover, active states, icon stroke on card hover |

These three values are the complete palette. No gradients, no opacity washes of other colors,
no additional brand colors may be introduced without an explicit constitution amendment.

#### Card & Panel Style (the signature element)

Cards, panels, modals, and any framed UI container must follow this exact specification:

- **Background:** `var(--cream)` — same as the page, no contrast fill
- **Border:** `2.5px solid var(--ink)` — full perimeter, all four sides
- **Border-radius:** `0` — absolutely no rounded corners anywhere in the app chrome
- **Permanent hard shadow** (resting state) — 7 identical 1 px offset layers:
  ```css
  box-shadow:
    1px 1px  0 var(--ink),
    2px 2px  0 var(--ink),
    3px 3px  0 var(--ink),
    4px 4px  0 var(--ink),
    5px 5px  0 var(--ink),
    6px 6px  0 var(--ink),
    7px 7px  0 var(--ink);
  ```
- **Hover state** — card lifts by `translate(-4px, -4px)`, shadow extends to 11 layers with
  coral on the 11th:
  ```css
  transform: translate(-4px, -4px);
  box-shadow:
    1px 1px  0 var(--ink),
    2px 2px  0 var(--ink),
    3px 3px  0 var(--ink),
    4px 4px  0 var(--ink),
    5px 5px  0 var(--ink),
    6px 6px  0 var(--ink),
    7px 7px  0 var(--ink),
    8px 8px  0 var(--ink),
    9px 9px  0 var(--ink),
    10px 10px 0 var(--ink),
    11px 11px 0 var(--coral);
  ```
- **Transition:** `transform 0.15s ease, box-shadow 0.15s ease`

This stacked-shadow technique gives physical "paper thickness" depth without any blur or glow.
Blurred box-shadows, drop-shadows, and glassmorphism effects are **explicitly prohibited**.

#### Buttons

All buttons (primary actions, secondary actions, toggles) follow the same card rules:
- Border: `2.5px solid var(--ink)`
- Border-radius: `0`
- Background: `var(--cream)` at rest
- Hard shadow (resting): same 7-layer pattern as cards (scaled to button size — can reduce to
  4–5 layers for small buttons)
- Hover: `translate(-2px, -2px)` + extended shadow + coral final layer
- Active/pressed: `translate(2px, 2px)` + reduced shadow (simulates pressing the button down)
- Font: system font stack, uppercase or sentence-case, `letter-spacing: 0.03em` minimum
- No ghost buttons, no text-only buttons, no pill-shaped buttons

#### Typography

- **Font family:** `-apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif` for all UI chrome
  (navigation, buttons, labels, captions). Arabic content uses the Arabic system font stack.
- **Note content** (the rendered note body) retains the fonts from the note renderer
  (Quicksand, Literata, Cairo) as defined in `medicine-cloud-note(claude).html` — these are
  content fonts, not UI chrome fonts.
- **Body text:** `color: var(--ink)`, default weight
- **Small labels and captions:** `font-size: 0.85rem`, `letter-spacing: 0.03em`
- No italic decorative headings in the UI chrome (italic is reserved for note content)

#### Icons

- All icons are **inline SVGs** — no icon font libraries, no PNG icon files
- `stroke: var(--ink)` at rest; `stroke: var(--coral)` on hover (when inside a hoverable card)
- `stroke-width: 2.5`
- `fill: none`
- `stroke-linejoin: round`, `stroke-linecap: round`
- Size: `70px × 70px` for card/feature icons; `20px × 20px` for inline UI icons

#### Prohibited Styling Patterns

The following are **explicitly banned** in all UI chrome components:
- Rounded corners (`border-radius` of any value other than `0`)
- Blurred box-shadows (`box-shadow` with blur radius > 0)
- Gradient backgrounds (linear-gradient, radial-gradient)
- Glassmorphism (backdrop-filter, rgba translucent panels)
- Animations longer than `0.2s` on interactive elements (the snap is part of the aesthetic)
- External icon libraries (Font Awesome, Material Icons, Heroicons, etc.)
- More than three colors in the palette (cream, ink, coral) — exceptions only by constitution amendment

#### Exceptions — Note Content Area

The rules above apply to **app UI chrome only** (navigation, cards, buttons, dialogs, dashboard,
editor, settings). The **note content area** (the `.note-content` div that renders the user's
actual study material) retains its own typographic and visual system from
`medicine-cloud-note(claude).html`. The note renderer's term link colors (green, blue, red, orange,
purple) are permitted inside `.note-content` as they are functional semantic indicators, not
brand decoration.

## Article 9 — Development Process (The Mandatory Loop)

> Every spec in specs.md is processed through this loop exactly once, in order.
> The next spec may not begin until the current one is DONE or CANCELLED.

### 9.1 The Loop

```
FOR EACH SPEC (in order from specs.md):

  STEP 1 — SPEC REVIEW (no code written yet)
    a. Read the full spec in detail.
    b. Check against the constitution. Conflict? Revise the spec.
    c. Check compatibility with all previously DONE specs. Conflict? Resolve.
    d. Check for sufficient detail. Too vague for an AI to build? Expand it.
    e. Mark spec status: APPROVED.

  STEP 2 — BUILD
    a. Implement exactly what the spec describes. Nothing more.
    b. Do not skip any described sub-components or acceptance criteria.
    c. No scope creep. Future ideas go into specs.md as new entries.

  STEP 3 — VERIFY
    a. Test every acceptance criterion listed in the spec.
    b. Test: light theme AND dark theme.
    c. Test: mobile 360px AND desktop 1280px.
    d. Test: Arabic note (RTL) AND English note (LTR) rendering.
    e. Confirm all previously DONE specs still pass.

  STEP 4 — OUTCOME
    All pass  ->  Mark spec DONE. STOP. Wait for user command to proceed.
    Failure   ->  Attempt fix -> return to STEP 3.
    Can't fix ->  Mark spec BLOCKED. STOP. Notify user with full explanation.
```

### 9.2 Scope Discipline
During BUILD, nothing outside the spec's scope may be added. Future ideas go to the backlog.

### 9.3 Backward Compatibility
New specs must not break previously DONE specs. Verify explicitly in Step 3e.

### 9.4 Blocking, Not Abandoning
BLOCKED specs are never silently skipped. The project owner decides the next action.

## Article 10 — Code Quality & Maintainability

### 10.1 Intended File Structure

```
MC-WEB_APP/
  dev/
    constitution.md          <- this file
    specs.md                 <- spec registry
  src/
    index.html               <- app shell / entry point
    css/
      base.css               <- design tokens, resets, base styles
      templates/
        classic.css          <- "classic" template tokens
        <name>.css           <- additional templates
    js/
      app.js                 <- bootstrap / router
      auth.js                <- Supabase auth logic
      notes.js               <- note CRUD operations
      renderer.js            <- JSON-to-HTML rendering (media tags, term links)
      export.js              <- HTML export logic
      pdf.js                 <- PDF export (calls serverless function)
      templates.js           <- template loading / switching
    pages/
      login.html             <- login / access gate
      dashboard.html         <- note library / home
      editor.html            <- paste JSON -> preview -> save
      note.html              <- single note reader (handles share links too)
      settings.html          <- user settings
    functions/
      pdf-export.js          <- Cloudflare Worker: headless Chrome PDF
  .env.example               <- committed: empty placeholder values
  .env                       <- NOT committed: real secrets
  .gitignore                 <- must include .env
```

### 10.2 Readability
Non-obvious implementation choices must have inline comments explaining the reasoning.
This especially applies to: the rendering pipeline, RLS policies, and the PDF export function.

### 10.3 Secrets
- Supabase URL + anon key: `.env`, injected at build time.
- Supabase service-role key: never in any frontend file.
- `.env` is always in `.gitignore`.

### 10.4 Git Commit Convention
Every completed spec gets a commit:
  [SPEC-N] short description of what was built
